# UTF-8 ⇄ CP932 化け文字メモ（マッピング表の根拠）

`cp932-mapping.json` の設計根拠と、表を拡張する際の判断材料。

## なぜ正規化が要るか

CP932（Windows-31J / Microsoft の Shift_JIS 拡張）と Unicode の往復には、**同じ見た目の文字が
複数の Unicode コードポイントに対応する**ために起きる古典的な「丸め」問題がある。開発側（UTF-8）で
入力した文字が CP932 に**そのままでは存在しない／別コードに丸められる**ため、変換前に「CP932 で
表現できる正規の和字」へ寄せる必要がある。

参考: <https://www.tohoho-web.com/ex/dash-tilde.html>

### 1. 波ダッシュ・全角チルダ問題
- `〜` U+301C（WAVE DASH）は CP932 では `0x8160` に**エンコードできる**が、その `0x8160` を
  **decode すると `～` U+FF5E（FULLWIDTH TILDE）として返る**（同一バイト列に両者が対応）。
  → つまり `〜` は round-trip で黙って `～` に化ける（**ロッシー**）。
- これを放置せず、変換前に `〜 → ～` を**明示的に確定**させて挙動を決定論的にする。
- （実測: `'〜'.encode('cp932')` = `81 60` → `b'\x81\x60'.decode('cp932')` = `～`）

### 2. バックスラッシュ・円マーク問題
- CP932 の 0x5C は ASCII バックスラッシュ `\` だが、日本語フォントでは円マーク `¥` に見える。
- `.ps1` / `.bat` では `\`（パス区切り）を**構文として使う**ため、`\` U+005C は**不変**にする。
- `¥` U+00A5 を `\` に化けさせる/その逆は厳禁。→ マッピング表に `\` `¥` を含めない（保護）。

### 3. チルダ・オーバーライン問題
- `~` U+007E（ASCII TILDE）と `‾` U+203E（OVERLINE）の混同。
- `~` はスクリプトで使うため**不変**。OVERLINE 系も ASCII を維持。

### 補足: 半角記号に「決定不能性」は無い（codec 実挙動で実証）

「Win→UTF-8 後に `\` をそのまま残してよいか決められないのでは」という懸念は不要。`cp932` コーデックは
ASCII 域で**決定論的な全単射**であり、見た目の `¥`/`‾` は JIS X 0201 系フォントの**描画**であって
コードポイントの差ではない（実測）:

| 検証 | 結果 |
|---|---|
| `0x5C` decode | 必ず `U+005C` `\`（`¥` にならない） |
| `0x7E` decode | 必ず `U+007E` `~`（`‾` にならない） |
| `\` `~` encode | `5c` / `7e`（往復安定） |
| `¥` U+00A5 encode | `UnicodeEncodeError` → `--to-win` の NG パスが検出・exit 1（黙って `\` 化しない） |
| `‾` U+203E encode | `UnicodeEncodeError` → 同上 |
| `￥` U+FFE5 / `￣` U+FFE3 encode | `81 8f` / `81 50`（正規の CP932 2バイト・正常） |

→ `\` `~` は **`--to-unix` 後も不変が唯一の正解**。本物の半角円・オーバーラインが UTF-8 ソースに紛れても
サイレント破損はなく、必ず報告されて作業者が判断できる（全角 `￥`/`￣` を使うか除去するか）。

## 既定マッピング（`to_win` = UTF-8→CP932 で正規化）

| 開発側(UTF-8) | → | Windows側(CP932 で正規) | 種別 |
|---|---|---|---|
| `〜` U+301C 波ダッシュ | → | `～` U+FF5E 全角チルダ | round-trip ロッシーの確定化（encode 可・decode で `～` 化） |
| `−` U+2212 マイナス | → | `－` U+FF0D 全角ハイフンマイナス | round-trip ロッシーの確定化（encode 可・decode で `－` 化） |
| `—` U+2014 EM ダッシュ | → | `―` U+2015 水平バー | **encode 不可の回避**（U+2014 は cp932 に存在せず UnicodeEncodeError） |
| `‖` U+2016 二重縦線 | → | `∥` U+2225 平行記号 | round-trip ロッシーの確定化（encode 可・decode で `∥` 化） |
| `¢` U+00A2 / `£` U+00A3 / `¬` U+00AC | → | `￠` / `￡` / `￢`（全角） | round-trip ロッシーの確定化（encode 可・decode で全角化） |

> 実測（Python 3.x cp932）: `〜→8160`/`−→817c`/`‖→8161`/`¢→8191`/`£→8192`/`¬→81ca` は
> いずれも encode 可だが decode で全角形に化ける（ロッシー）。`—`(U+2014) のみ encode 不可。

## 保護（変換しない＝スクリプト構文を壊さない）

- `\` U+005C バックスラッシュ（`¥` に化けさせない）
- `~` U+007E チルダ、`‾` U+203E オーバーライン関連
- その他すべての ASCII 記号

## 逆方向（`to_unix` = CP932→UTF-8）の扱い

- Python の `cp932` デコードは**ロスレス**なので、復号自体に化けは起きない。
- `to_unix` 写像（`～ → 〜` 等）は **`--restore` 指定時のみ**適用する任意機能。
- **既定では逆変換しない**: `－`（全角ハイフン）や `―`（水平バー）は正規の日本語文中にも
  普通に現れるため、無条件に `−`（マイナス記号）等へ戻すと**正しい和字を誤変換**してしまう。
  自分が `--to-win` した直後に完全な round-trip を戻したい時に限り `--restore` を使う。

## 表の拡張方法

`--to-win` 実行時に CP932 エンコード不可文字があると、スクリプトが
`line N: '文字' U+XXXX` 形式で列挙して非ゼロ終了する。出た文字について
「CP932 に存在する正規の対応文字」を調べ、`cp932-mapping.json` の `to_win`
（必要なら `to_unix` 逆写像も）に追記して再実行する。黙ってデータが消えることはない。
