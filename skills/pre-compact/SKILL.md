# pre-compact（コンパクション前事前準備）

`/compact` または `/clear` 実行前に呼び出し、次セッション再開時に最低限の文脈が復元できる状態を整える Skill。**最初に終了方式を判定し、その方式で必要な準備だけを行う**ことで、不要なメモリ退避を避けつつ確実に再開可能な状態を作る。

## 出力方針

**未コミット変更（あれば）と、最終的な終了方式の提示（推奨方式＋選択肢、`/compact` 選択時はコメント案）のみを出力する。** 方式判定の現状把握（ステップ 1）と方式別準備（ステップ 3）は完全 silent。コンテキスト消費を抑えるための方針。

**silent ステップでの禁止事項**: 「サイレントで実行中」「ステップ N を処理中」「メモリを確認しています」等の進捗説明・確認メッセージ・実行状況報告を含め、**いかなるテキストも出力してはならない**。完全無言で処理する。

## 実行手順

**方式を先に決めてから、その方式に必要な準備だけを行う**。順番に実施する。

### 1. 終了方式の判定（silent）

会話コンテキストから現状を把握し（**この段階ではファイルに書き込まない**）、どちらの方式を推奨するか決める。**出力しない**。

把握する観点:

- 現在実施中のタスクと中断箇所、次セッションの最初のアクション
- 次アクションが具体的に確定しているか（ファイルだけで再開できるか）
- シリアライズしにくい暗黙知（探索的設計の判断経緯・デバッグの試行錯誤の軌跡・却下した選択肢の理由など）が次の手がかりとして必要か
- 保留中の判断事項（作業指示者が決める必要があるもの）

判定基準:

- **`/clear` 推奨**: 次アクションが具体的に確定しており、継続情報を memory ファイルに落とし切れる。暗黙知への依存が小さい。会話履歴を破棄してもファイルだけで再開できる場合（コンテキスト消費が最小）。
- **`/compact` 推奨**: 落とし切れない暗黙知が次の手がかりに必要で、ファイルに落とし切れない。会話要約を新コンテキストに残す。

判定が曖昧な場合は、迷わず **`/clear` 側の準備（全量外部化）を採る**（外部化は `/compact` でも無害なため安全側）。

> 補足: `/clear` でもルート/.claude CLAUDE.md・MEMORY.md・ツールスキーマ・MCP instructions 等の固定ベースラインは再ロードされる。`/clear` の節約効果は実質「会話要約＋直近メッセージ分」であり、ゼロにはならない。

### 2. 未コミット変更の確認（方式共通）

`git status` で未コミット変更を確認する。

- **変更なし**: 出力しない
- **変更あり**: コミットメッセージ案と対象ファイルを提示し、作業指示者の判断を仰ぐ（この場合のみ出力する）

### 3. 方式別の準備（silent）

ステップ 1 の判定に応じて、**必要な準備だけ**を行う。**出力しない**。

#### 3A. `/compact` 系（退避は最小限）

会話要約が新コンテキストに残るため、継続情報の全量外部化は不要。要約が**運べない／取りこぼす恐れのある長期事実だけ**を memory に反映する:

- ユーザの嗜好・恒久的な合意事項・プロジェクトのマイルストーン等、セッションを越えて効く事実
- 進行中タスクの細かな途中経過は会話要約に委ね、ここでは書かない（重複退避を避ける）

#### 3B. `/clear` 系（全量外部化）

会話は完全破棄され、再開は memory ファイルのみに依存する。継続情報を取りこぼさず memory に外部化する:

1. 進行中タスクの最新ステータス・中断箇所・次アクション・保留事項を該当 memory ファイルへ反映する。
2. 再開の起点は `MEMORY.md` の索引行（フック＝各行の `—` 以降の説明文）。今回の継続タスクのフックが次の 3 要素を**すべて**含むか判定し、欠ける場合は Edit でフック行を更新する: (a) タスクが未完了・進行中である旨、(b) 再開時の最初のアクション、(c) 該当 memory ファイルを読む指示。
3. フックは過去の達成サマリではなく「次に何をするか」を指す**現在形**の文にする。

### 4. 終了方式の提示

推奨方式を提示する。判定が曖昧だった場合は両条件への適合度を 1〜2 文で述べ、推奨を一方に決めた上で両方を選択肢として示す。最終決定は作業指示者が行う。

#### 4A. `/compact` を提示する場合

`/compact <指示>` に渡すコメントを生成して提示する。

**コメントの書き方（厳守ルール）**:

- **英語で書く（厳守）**: `Focus on …` / `Preserve …` の構文を用いる。タスク名・ファイル名等の固有名詞のみ日本語のままでよい。公式例 `Focus on the API changes` に倣う。
- **150 字以内（厳守。`/compact ` を除く指示文本文）**: コンパクターは intent を解釈するため、詳細なタスクリストより端的な保存指示が有効。**特に冗長化しやすいので、超過したら 150 字以内に削るまで再修正してから提示する**。
- **「Focus on X」「Preserve Y」フレーミングで intent を伝える**: 何を中心に進めるか・何を保持するか。
- **ファイルパス・指摘 ID（C1 / I1 等）・手順の具体的羅列を避ける**: コンパクターが会話から自動抽出するため冗長になる。

**盛り込む観点**（優先順位順）: ①現在進行中のタスク（最優先） ②未完了の残ステップの概要 ③保留中の判断事項（あれば） ④未コミット変更（あれば）。

**セルフチェック**（出力前に必ず確認。1 つでも不適合なら修正してから提示）:

- [ ] 英語で書かれているか（固有名詞を除く）
- [ ] 指示文本文が 150 字以内か
- [ ] 「Focus on / Preserve」フレーミングを使っているか
- [ ] パス・指摘 ID・手順の羅列を避けているか

**出力形式**（作業指示者がそのままコピーして使えるよう提示）:

```
【/compact コメント提案】
/compact <生成した指示コメント>
```

> `/compact` の代わりに `/clear` を選びたい場合はその旨を伝えてほしい。会話要約が残らないため、先に継続情報を memory へ外部化（ステップ 3B）してから `/clear` する。

#### 4B. `/clear` を提示する場合

ステップ 3B でフックを整備済みであることを前提に、「`/clear` 実行後は `MEMORY.md` のフック行を起点に該当 memory ファイルを読んで再開する」旨を伝える。再開時は当該 memory ファイルのみを読み、無関係な大型ファイルを無闇に読み直さない。

## 完了確認

ステップ 4 の出力後、選択された方式に応じて作業指示者に伝える:

- `/compact` 選択時: 「上記コメントを使って `/compact` を実行してください」
- `/clear` 選択時: 「`MEMORY.md` のフックを再開キーとして整えました。`/clear` を実行してください」

## /compact コメントに関する公式情報

参照元: Claude Code 公式ドキュメント（`/compact` の項。ローカル DL 済みの場合は `llms-full.txt` を Grep）

> For more control, run `/compact <instructions>`, like `/compact Focus on the API changes`

> Customize compaction behavior in CLAUDE.md with instructions like `"When compacting, always preserve the full list of modified files and any test commands"` to ensure critical context survives summarization

コメントは自由記述でコンパクターが intent を解釈して処理する。常時保持したい保存指示は CLAUDE.md に書く方法もある。

## 設計背景

`/compact` は built-in コマンドであり `PreCompact` hook で前処理を挟む方法もあるが、hook は Claude のコンテキスト外（シェル/別 LLM 呼び出し）で動作するため、現在の会話コンテキストを踏まえた Claude 自身による能動的な準備は実行できない。本 Skill が会話コンテキスト内で動作する（hook ではなく明示呼び出しの Skill とした）のはこのためである。
